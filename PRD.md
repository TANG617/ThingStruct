# ThingStruct PRD

- 版本：Draft v0.2
- 日期：2026-03-25
- 状态：重构草案
- 文档目的：基于当前实现与产品方向，重新明确 ThingStruct 的真正核心、边界与未来演进方式

## 1. 一句话定义

ThingStruct 不是待办清单，也不是日历。

ThingStruct 是一个单人使用的、以 routine 为核心的「日常运行系统」：用户每天先选择今天要运行哪一种生活模板，随后 App 在一天中持续、温和地提示当前所处状态与下一步 checklist，而不是要求用户不断创建新任务。

## 2. 产品本质

ThingStruct 的本质不是“帮用户记录事情”，而是“帮用户运行一种日常结构”。

它服务的不是：

- 零散、一次性的事务管理
- 项目管理
- 每天不断新增事项的 inbox 工作流
- 日历事件同步和调度

它服务的是：

- 重复出现的生活模式
- 稳定的工作日结构
- 在特定情境下反复出现的 checklist
- 对自己说的固定提示语
- 一天中不同状态之间的平滑切换

用户每天做的核心动作不是“整理今天有哪些事”，而是“决定今天运行哪种模板”。

## 3. 产品定位

### 3.1 产品类别

ThingStruct 位于以下类别的交叉点：

- structured lifestyle app
- routine operating system
- day-structure runner
- calm execution tool
- checklist-based daily companion

### 3.2 与常见产品的根本区别

大多数效率产品的底层逻辑是：

- 用户不断添加任务
- 系统帮助排序、归类、提醒
- 今天要做什么，取决于任务列表里新增了什么

ThingStruct 的底层逻辑是：

- 用户先定义几种可复用的“一天”
- 每天选择今天运行哪一种
- 一天中的提醒、任务、备注都从这个结构里自然出现
- 手机端主要负责展示与执行，而不是持续创建

换句话说：

- todo app 的核心是“收集事项”
- ThingStruct 的核心是“运行结构”

## 4. 产品哲学

### 4.1 单人绝对成立

ThingStruct 是绝对单人的产品。

这意味着：

- 不会有多人共享
- 不会有协作编辑
- 不会有团队视图
- 不会有评论、分配、同步给别人

即便未来支持导出或云端编排，那也只是为了更好地服务“同一个人”的复盘和辅助，不改变单人本质。

### 4.2 本地优先，但允许云端编排

ThingStruct 不做传统意义上的云同步。

本地始终是唯一业务真相来源。云端可以存在，但只能扮演以下角色：

- 生成建议
- 编排复杂任务结构
- 产出模板草案
- 产出可导入的结构化内容

云端不能直接成为产品真相。

正确的模型是：

- 云端负责“建议和编排”
- 本地负责“确认、落地、执行、持久化”

这使 ThingStruct 可以天然与 AI 配合，但不会演变成一个依赖远程同步的 SaaS。

### 4.3 routine-first，而不是 task-first

ThingStruct 优先描述的是 routine，不是一次性的任务。

例如适合出现在 ThingStruct 中的是：

- 出门前带水杯
- 下班前收桌面
- 午饭后散步 10 分钟
- 睡前准备明天的衣服
- 健身前做热身

而不适合出现在 ThingStruct 中的是：

- 今天 3 点见客户
- 下周交报告
- 给某某发邮件
- 跟进某个项目

如果某种一次性事件真的会改变一天的结构，它应当以“模板切换”或“另一天的结构选择”进入系统，而不是以一个临时 todo 混入系统。

### 4.4 手机端是“运行终端”，不是“编排终端”

ThingStruct 的手机端重点不是创建，而是运行。

端侧应该强调：

- 看现在
- 勾 checklist
- 看接下来
- 轻微修正
- 选择今天模板

端侧应弱化：

- 大量新建
- 复杂编排
- 高频录入
- 把手机变成一个随时写入新任务的入口

创建能力仍然保留，但应当是粗糙的、克制的、支撑性的，而不是产品重心。

### 4.5 提醒是配合，不是打断

ThingStruct 的提醒不应像闹钟那样中断用户。

它更像是一种配合式提示：

- 轻量出现
- 不抢夺焦点
- 不强制中断当前动作
- 在系统表面持续可见
- 更像一个与用户配合的副驾驶，而不是一个命令式的监督者

## 5. 目标用户

### 5.1 核心用户

- 有稳定生活节律的人
- 希望把工作日过成“有结构的一天”的人
- 重视 routine 胜过追求任务清空的人
- 不希望每天在手机上反复输入新任务的人
- 需要柔和提醒与明确上下文的人

### 5.2 非目标用户

- 想把它当成通用 todo list 的用户
- 需要团队协作和任务流转的用户
- 主要管理项目型工作而非生活节律的用户
- 需要跨平台同步数据库的人
- 主要依赖外部日历来驱动一切的人

## 6. 典型使用场景

### 6.1 工作日模板选择

用户每天开始时，先选择今天运行哪一种模板：

- 标准工作日
- 周二加班模板
- 周五早下班模板
- 居家办公模板
- 恢复日模板

模板一旦选定，整天的结构、提醒、checklist、备注提示都随之出现。

### 6.2 日常执行

用户在一天中更多时候不是“规划”，而是：

- 看一眼当前 block
- 看当前应该做的 checklist
- 勾掉已经完成的固定步骤
- 接受一个轻量、非打断的提醒

### 6.3 少量修正

如果今天和模板有一点点不同，用户可以：

- 调整某个时间块
- 临时替换一个 block
- 少量编辑备注或 checklist

但这应当是次要行为，不应成为日常主流行为。

### 6.4 外部智能编排导入

未来用户可在云端或 AI 工具中完成更复杂的编排，然后导入本地：

- AI 根据用户目标生成一版工作日模板
- AI 输出一版新的 checklist 组合
- 用户在本地预览、确认并应用

这是一种“本地执行，云端协助”的结构，而不是云端同步。

## 7. 核心产品原则

- 单人优先：所有功能都围绕“一个人如何过好自己的一天”展开。
- 模板优先：先选今天运行哪种结构，再进入执行。
- 展示优先：移动端首先负责展示与引导，而不是录入。
- routine 优先：固定模式比临时事务更重要。
- checklist 优先于 todo：任务是动作清单，不是事项收集箱。
- 温和提醒：提醒是配合式、伴随式，而不是侵入式。
- 本地为真：所有已确认结构最终落在本地文档上。
- 云端辅助：云端可以建议，但不能直接替代本地真相。

## 8. 产品不是什么

ThingStruct 不是：

- 待办应用
- GTD 工具
- 项目管理工具
- 外部日历外壳
- 习惯打卡产品
- 团队协作产品
- 多人共享家务板
- 以快速创建任务为核心的手机工具

## 9. 产品是什么

ThingStruct 是：

- 一个“今天运行哪个版本的生活”的选择器
- 一个把一天表达为状态结构的系统
- 一个在正确时间显示正确 checklist 的陪伴工具
- 一个让重复生活模式可以被设计、复用、执行的系统
- 一个把“展示”和“现在该做什么”做到足够清晰的日常终端

## 10. 核心概念定义

### 10.1 DayPlan

某一个本地自然日的完整运行计划。

### 10.2 TimeBlock

一天中某一段状态。它不是日历事件，而是“我这段时间处于什么状态”。

### 10.3 BaseBlock

构成一天底层骨架的 block，例如起床、上午工作、午餐、晚间恢复。

### 10.4 OverlayBlock

叠加在更高层的状态，例如“深度工作”、“周会”、“健身”、“出门准备”。

### 10.5 activeChain

某个时刻真正处于生效状态的一条父子链。

### 10.6 activeBlock

当前生效链条中最高层的那个 block。

### 10.7 taskSourceBlock

当前应该承接未完成 checklist 的 block。

这点非常关键：

- 用户当前所处状态，和当前任务应从哪一层出现，不一定是同一个概念。

### 10.8 Task

ThingStruct 中的 task 不是“今天有哪些事”，而是某个状态里的 checklist。

正确示例：

- 出门前带门禁卡
- 午休前收电脑
- 训练后拉伸

错误示例：

- 今天完成投标方案
- 给客户打电话

### 10.9 Note

ThingStruct 中的 note 更像“固定地对自己说的话”，是一个在某种状态下需要反复被看见的提示语。

例如：

- 午饭别吃太快
- 训练前先热身
- 晚上不要再开新工作

它不是临时记录，不是流水日志，不是会议纪要。

### 10.10 Template

Template 不是抽象日历模版，而是“一种可运行的一天”。

## 11. 核心生活逻辑

ThingStruct 的真正核心流程应当是：

### 11.1 先选今天

每天先决定今天属于哪种日子：

- 普通工作日
- 加班日
- 早下班日
- 休息日
- 健身强化日

### 11.2 再运行一天

一旦选择模板，用户就不需要不断补充任务，而是沿着这一天的结构前进。

### 11.3 提示不是命令

当某个时间点到了，ThingStruct 不是强制打断用户，而是：

- 在 `Now`
- 在 Widget
- 在 Live Activity
- 在系统提醒

轻轻告诉用户：

- 现在到了哪个状态
- 该注意什么
- checklist 里还有什么

### 11.4 一天结束后沉淀结构

如果今天的结构不错，用户可以：

- 作为新模板保存
- 更新既有模板
- 将其作为之后某种日子的变体

## 12. 主要产品目标

- 帮助用户每天快速决定今天运行哪一种模板。
- 帮助用户在一天中始终知道自己处于哪种状态。
- 帮助用户在合适的时刻看到合适的 checklist。
- 让提醒尽量以温和方式出现，而不是成为中断源。
- 让手机端成为一个优秀的执行终端，而不是繁杂的录入终端。
- 让模板成为日常生活逻辑的真正核心。
- 为未来 AI 编排保留可导入接口，但不放弃本地真相。

## 13. 主要非目标

- 不做团队协作。
- 不做共享任务空间。
- 不做项目级事务管理。
- 不做通用待办收集箱。
- 不做外部日历集成。
- 不做传统云同步。
- 不做“手机上随时记一条任务”的主路径。
- 不做高侵入性的闹钟式提醒系统。

## 14. 信息架构

ThingStruct 仍采用三个一级页面，但产品重心需要重新解释。

### 14.1 Now

定位：

- 今日运行终端
- 当前状态查看器
- checklist 执行面板

优先回答：

- 现在是什么状态？
- 我现在该做什么？
- 我现在该记住什么？

### 14.2 Today

定位：

- 今日结构查看器
- 轻编辑器
- 模板运行结果的时间轴投影

优先回答：

- 今天整体结构是什么？
- 当前在哪个位置？
- 如果今天与模板略有不同，该怎么轻微修正？

### 14.3 Library

定位：

- 模板库
- 日结构资产库
- 导入导出与系统设置中心

优先回答：

- 我有哪些“生活模板”？
- 今天和明天要运行哪一个？
- 如何导入外部编排结果？

## 15. 功能需求

### 15.1 每日模板选择

这是产品最核心的功能之一，且应在 PRD 中被提高到中心地位。

功能定义：

- 用户每天应能明确选择“今天运行哪一种模板”。
- 这个选择应是显式的、可理解的、低成本的。
- 模板选择应直接决定当天的 block 结构、checklist、note、提醒。
- 用户可以为特殊日子手动切换模板，而不必依赖复杂规则。
- 可保留 weekday rule 作为辅助，但它不应掩盖“每天选择今天”的核心心智。

产品要求：

- 模板的差异必须对用户可读，而不是只有名字不同。
- 用户应能快速理解“标准工作日”和“周二加班日”到底区别在哪。

状态：

- 当前已有模板与 schedule 体系
- 产品重心需要进一步从“规则分配”转向“每日显式选择”

### 15.2 Now：当前运行视图

功能定义：

- 根据当前时刻解析 `activeChain`。
- 展示当前链条中的 note 与 checklist。
- 将未完成 checklist 作为绝对焦点。
- 上层已完成时，任务来源自动下沉到更低层。
- 支持快速勾选 checklist。
- 明确表达当前 blank 时段或无行动项的状态。
- 保持信息简短、明确、即时。

产品要求：

- `Now` 必须是使用频率最高的页面。
- `Now` 的语言应该像“陪伴你运行这一天”，而不是“提醒你还有很多事没做”。

状态：

- 已实现

### 15.3 Today：结构查看与轻编辑

功能定义：

- 以时间轴展示当天结构。
- 展示 base / overlay 的嵌套关系。
- 展示当前时间线。
- 支持查看和选择 block。
- 支持少量修正：新增粗糙 block、轻量编辑、resize、取消。
- 支持从空档补建真实 base block。

产品要求：

- `Today` 是“查看和修正”，不是“高频创作”。
- 任何复杂编排能力都应被弱化，不应把用户训练成每天在手机上搭积木。

状态：

- 大部分已实现

### 15.4 创建与编辑能力

产品立场：

- 创建存在，但不是核心。
- 编辑存在，但主要用于小幅修正。

功能定义：

- 允许用户创建 base block。
- 允许用户创建 overlay。
- 允许用户编辑 block 标题、时间、note、checklist、提醒。
- 允许用户取消 block。
- 允许用户保存后立即更新当天运行结构。

产品要求：

- 端侧创建流程可以粗糙，但必须可靠。
- 不以“快速新建任务”作为主卖点。
- 更推荐“从已有模板或昨天继承，再小幅修改”。

状态：

- 已实现，但在产品叙事上需要降权

### 15.5 Checklist 任务系统

功能定义：

- task 仅为 checklist item。
- task 绑定在具体 block 上。
- task 排序稳定。
- task 支持快速完成。
- task 完成后驱动 `taskSourceBlock` 重新计算。

产品要求：

- task 的语义必须始终接近“步骤清单”，而不是“事务列表”。
- 一次性、不重复的事项不应鼓励进入系统。
- 移动端不应强调快速新增 task。

状态：

- 已实现

### 15.6 Note 系统

功能定义：

- note 绑定在 block 上。
- note 在执行上下文中以低于 task 的权重显示。
- note 的语义更像固定的自我提示语。

产品要求：

- note 不应演变为长文本笔记系统。
- note 的最佳状态是短、稳定、可重复出现。

状态：

- 已实现

### 15.7 提醒系统

功能定义：

- 支持 block 开始提醒。
- 支持开始前提醒。
- 从当天结构自动推导提醒计划。
- 在通知和系统表面中展示当前状态与下一步。

产品要求：

- 默认提醒风格应尽量温和。
- 提醒应更接近“轻推一下”，而不是“强制中断”。
- Widget、Live Activity、通知、Now 页面应共同构成一个低侵入的提醒体系。

状态：

- 技术能力已实现
- 产品体验仍需继续朝“非侵入式”方向收敛

### 15.8 模板系统

模板系统是 ThingStruct 的真正心脏。

功能定义：

- 从真实日计划中生成候选模板。
- 保存为正式模板。
- 编辑正式模板。
- 为不同场景维护多种模板变体。
- 为今天和未来某一天选择模板。
- 在需要时 materialize 为当天的日计划。

产品要求：

- 模板是“一种生活版本”，而不是机械复制排班。
- 模板预览必须让用户看懂“这一天会怎么过”。
- 模板差异应主要体现在结构、状态、提醒和 checklist 上。

状态：

- 已实现

### 15.9 模板分配与每日选择

这是当前实现中需要重构产品叙事的关键部分。

功能定义：

- 用户可以显式为今天选择模板。
- 用户也可以保留 weekday rule 作为默认值。
- 当默认模板不合适时，用户应当很容易切换到另一个模板。

产品要求：

- “今天选哪种模板”应当是显性动作，不应完全隐藏在规则自动化里。
- 用户对模板切换的理解，应接近于“今天运行哪个版本的生活”。

状态：

- 当前已有 weekday rule / override
- 未来产品应强化“每日显式选择模板”的入口和心智

### 15.10 导入 / 导出

导入导出不是协作功能，而是单人复盘与外部编排接口。

功能定义：

- 导出今天结构用于个人复盘。
- 导入结构化 YAML 替换或更新本地内容。
- 导入前提供摘要预览。
- 导入后以本地文档为准。

产品要求：

- 导出应服务于个人整理与反思，而非共享协作。
- 导入应服务于外部编排结果落地，而非远程同步。

状态：

- 已实现

### 15.11 云端编排接口

这是未来产品的重要扩展方向，但必须明确边界。

功能定义：

- 允许云端或 AI 工具生成结构化日计划或模板草案。
- 支持这些结果被导入本地。
- 支持在本地预览、确认、应用。

明确边界：

- 不做传统云同步。
- 不做远程数据库作为真相源。
- 不允许远程直接无确认覆盖本地当天运行结果。
- 云端是“编排器”，本地是“执行器”。

产品意义：

- 这让 ThingStruct 可以天然接受 AI 辅助。
- 同时不破坏其单人、本地、可控的本质。

状态：

- 导入导出基础已具备
- 云端编排接口仍属于未来扩展

### 15.12 Widgets

Widget 是 ThingStruct “温和陪伴”的关键系统表面。

功能定义：

- 展示当前 block。
- 展示少量 checklist。
- 展示剩余行动项。
- 支持直接勾选。
- 点击回到正确上下文。

产品要求：

- Widget 不应像待办小组件那样变成任务板。
- 它应更像“今天的运行状态窗”。

状态：

- 已实现

### 15.13 Live Activities

Live Activity 是“当前这件事正在进行”的持续展示。

功能定义：

- 展示当前 block。
- 展示一个最关键的当前 task。
- 展示剩余数量。
- 提供一个轻量完成动作。

产品要求：

- Live Activity 是陪伴式、持续式，不是高刺激式。
- 它应该帮助用户维持当前状态，而不是制造存在感。

状态：

- 已实现

### 15.14 Controls / App Shortcuts / Quick Actions

这些系统入口的共同定位是：

- 快速进入当前运行状态
- 快速完成当前 checklist
- 快速打开当前 block

产品要求：

- 所有系统入口都应围绕“现在”展开，而不是围绕“新建”展开。
- 它们服务的是执行，而不是编排。

状态：

- 已实现

## 16. 信息优先级

ThingStruct 的信息优先级应当始终是：

1. 我现在在哪种状态里？
2. 我此刻该注意什么？
3. 我此刻还有哪几个 checklist 没完成？
4. 今天整体结构是什么？
5. 今天运行的是哪一个模板？
6. 只有在必要时，才是如何编辑它

## 17. 交互风格要求

### 17.1 展示优先

- 首屏优先展示当前运行状态。
- 减少需要思考的编辑入口。
- 把编辑留到真正必要时。

### 17.2 轻量完成优先

- 勾选 checklist 必须很快。
- 进入 App 后应几乎不需要思考就能完成当前动作。

### 17.3 创建降权

- 不鼓励用户随时打开手机加新任务。
- 不把“快速输入”当主能力。

### 17.4 温和提醒

- 不使用 alarm 式心智作为产品主轴。
- 鼓励通过系统表面“持续可见”来替代“强打断”。

## 18. 成功指标

ThingStruct 的成功不应以“新增了多少任务”衡量，而应以以下指标衡量：

- 用户是否每天会选择今天模板
- 用户是否高频使用 `Now`
- 用户是否在 App 外通过 Widget / Live Activity / Control 完成 checklist
- 用户是否逐渐形成多个稳定模板
- 用户是否减少了在手机上临时组织生活的负担
- 用户是否感受到提醒是“配合的”，而不是“打断的”

## 19. 风险

- 如果产品表达不清，用户可能误以为它只是另一个待办 App。
- 如果创建入口过重，会背离“运行终端”的方向。
- 如果模板选择不够显性，用户会把核心逻辑误解成自动排班工具。
- 如果提醒做得太强，会违背产品的陪伴式定位。
- 如果云端编排边界不清，会慢慢滑向远程真相和伪同步。

## 20. 当前实现与目标方向的关系

### 20.1 已经符合方向的部分

- 单人本地模型
- `Now` 作为执行页
- `Today` 作为时间轴结构页
- 分层 block 与 `taskSourceBlock`
- 模板系统
- Widget / Live Activity / Controls / Shortcuts
- 导入导出
- 通知动作

### 20.2 需要在产品层继续收紧的部分

- 明确把 sample data 从“产品的一部分”改为仅开发/过渡手段
- 明确把“端侧创建”降级为次要能力
- 强化“每日选择模板”作为第一动作
- 弱化“这是一个效率工具”的外部印象
- 继续把 task 语义收紧为 checklist
- 继续把 note 语义收紧为固定提示语
- 继续把提醒体验收紧为非侵入式

## 21. 最终产品主张

ThingStruct 想提供的不是“更多效率”，而是“更少犹豫地过一天”。

用户每天不需要在手机上重新规划人生。

用户需要的是：

- 早上决定今天是什么样的一天
- 白天在合适时刻被温和地提醒
- 当下知道自己处于什么状态
- 勾掉那些真正该在那个状态里完成的 checklist
- 逐渐把自己的生活沉淀成几种稳定、可运行的模板

如果说别的 App 是帮用户“管理事情”，ThingStruct 更像是在帮用户“运行生活”。
